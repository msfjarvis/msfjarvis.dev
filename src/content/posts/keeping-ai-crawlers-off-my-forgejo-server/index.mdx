> [!NOTE]
> This post will mention Cloudflare. People have a lot of strong opinions about the company's effective monopoly, myself included, but it was the fastest way for me to solve my immediate problem.
> I'm also employed at Cloudflare, and get a free Enterprise account from them, but I believe nothing I have set up here requires one. If someone tries this out and runs into any free-tier limitations, please let me know and I will make a correction.

Beginning 2026-06-18 I was starting to see a lot of slowness in my Forgejo instance, which I hoped would eventually subside, so I chose to just watch the load dashboards and wait.

The crawlers obviously did not relent, so I started earnestly mitigating it on 2026-06-25 by enabling fail2ban which unfortunately did not do much to help the situation. I had to take a hammer to the problem and put in a Cloudflare [security rule](https://developers.cloudflare.com/security/rules/) banning the countries of China and Vietnam where most of the requests were originating. The rule also carried a specific ban for accesses to the `oneplus3` repository because it was surprisingly a "fan" favorite. The Cloudflare block immediately got to work and within 16 hours mitigated over 130K hits.

However, the site's availability started dropping over the next few hours and I started seeing a near constant stream of `HTTP 521` responses which indicates Cloudflare couldn't reach my server. Turns out the log transformer I enabled for fail2ban to work wasn't logging the real client IPs, it was logging Cloudflare IPs, so fail2ban had just been banning Cloudflare 😬

Solving that wasn't too difficult, Caddy has the necessary config knobs to mark [reverse proxies as trusted](https://caddyserver.com/docs/caddyfile/options#trusted-proxies) and [forward the client IP headers](https://caddyserver.com/docs/caddyfile/options#client-ip-headers) correctly. I unbanned all the IPs in fail2ban, and after that I followed [this example](https://github.com/fail2ban/fail2ban/blob/2c1eb69381796c49e8866bb26fdc057bca159bfd/config/action.d/cloudflare.conf) to make fail2ban directly block offenders via Cloudflare's IP Access Rules.

Within 3 hours I was up to 2200 individual IPs banned, so I decided to do some analysis of banned IPs to figure out if there's a broader block I can instate and prevent running into the 50,000 limit. Most of them were popular consumer providers like Comcast and Verizon, so I banned the ASNs south of them:

```
- AS4229 Zenlayer
- AS150436 Byteplus
- AS62240 Clouvider
- AS62651 Strong Technology
- AS32934 Facebook
```

This helped to some degree but not _quite_, and I was still getting a sustained 50 to a 100 RPS which wasn't impossible to serve, just very wasteful and causing other things to slow down on this rather weak VPS. After around 12 hours, I realized fail2ban had now banned 31K individual IPs which was coming dangerously close to the 50K IP Access Rules limit from Cloudflare, and was admittedly a rather naive way of addressing the problem.

I turned off fail2ban, and started poring through my Forgejo logs to find patterns I could let Cloudflare handle via a [custom ruleset](https://developers.cloudflare.com/ruleset-engine/custom-rulesets/). This took a bit to figure out, but ultimately I landed on these, in order:

1. Block China and Vietnam. These still have the worst of the residential proxy botnet problem.
2. Block the known-bad ASNs from before.
3. Block anyone accessing the `/oneplus3` and `/revanced-patches` repositories since they had become super attractive scraper honeypots for some reason, I don't really need either to be available so this was fine.
4. Block fetches of `.bundle` archives, since I don't use them and they're expensive to generate.
5. Block (downgraded to Managed Challenge later) paginated history crawling:
   ```
   http.request.method eq "GET" and
   http.request.uri.path strict wildcard "/*/*/commits/commit/*" and
   http.request.uri.query contains "page="
   ```
6. Managed Challenge for all other anonymous history crawling
   ```
   http.request.method eq "GET" and
   not cf.client.bot and
   (
     http.request.uri.path strict wildcard "/*/*/blame/commit/*" or
     http.request.uri.path strict wildcard "/*/*/commits/commit/*" or
     http.request.uri.path strict wildcard "/*/*/src/commit/*" or
     http.request.uri.path strict wildcard "/*/*/blame/tag/*" or
     http.request.uri.path strict wildcard "/*/*/commits/tag/*" or
     http.request.uri.path strict wildcard "/*/*/src/tag/*" or
     http.request.uri.path strict wildcard "/*/*/find/tag/*" or
     http.request.uri.path strict wildcard "/*/*/commit/*.diff" or
     http.request.uri.path strict wildcard "/*/*/commit/*.patch" or
     (
       http.request.uri.path strict wildcard "/*/*/commit/*" and
       (
         http.request.uri.query contains "files=" or
         http.request.uri.query contains "show-outdated=" or
         http.request.uri.query contains "whitespace="
       )
     )
   )
   ```
7. Managed Challenge for reading historical copies of files
   ```
   http.request.method eq "GET" and
   not cf.client.bot and
   (
     http.request.uri.path strict wildcard "/*/*/raw/commit/*" or
     http.request.uri.path strict wildcard "/*/*/raw/tag/*"
   )
   ```

With all these in place the Git server is now _chilling_. I can't remember the last time it was serving so few spam hits that I could see the Prometheus `/metrics` query in the journal logs.

According to Cloudflare's logs it has mitigated 489.65K requests for me since I deployed this rule set, and of the 100K Managed Challenge that have been issued, only 3 have been solved. Go BOTS team!

Of course, I still had the problem of the now-useless IP Access Rules that fail2ban had deployed, which needed to be removed. Here I have some criticism of Cloudflare and the way the whole account/zone authentication situation works, because this took me over an hour to figure out and none of it was documented.

I tried using an account API token to fetch IP Access Rules, and consistently got 0 results from the API. I tried a user API token and got the same result. I switched from the `/account` API shape to the `/zone` one, and started getting the completely useless `code: 10000 message: Authorization error` error. No matter how I tried to call the API, I would either get 0 or an error, while I was staring at 31K of them in the dashboard. Finally, I decided to try the Global API key whose docs has a big [limitations section](https://developers.cloudflare.com/fundamentals/api/get-started/keys/#limitations) but no mention that it's the only way to access the supposed legacy IP Access Rules.

So now that I had wasted an hour and could _finally_ get back these 31K rules, I do a cursory analysis on them with MaxMind and obviously find no country or ASN I can do a simple ban on. To delete all these IP Access Rules I use the Cloudflare Python SDK to write a small script that would remove these serially since I didn't wanna go for max speed and then have to handle being rate limited. This took a whole 8 hours and 49 minutes to chew through all those IP Access Rules, but given I was effectively doing 1 RPS off my residential Internet connection I'm not mad about it.
