I chanced across this on one of my weeknotes, which seems to have already been deleted by Bluesky moderation or the poster by the time I got around to seeing it. I have been coasting along for a few months now without any moderation set up to easily moderate Webmentions, but I clearly should be paying more attention. This is far from being bad as far as spam goes, but I'd like to be prepared for the eventuality of something really terrible showing up in one of these.

So far I had been resorting to manually running queries in the Cloudflare dashboard's [D1 SQL Explorer](https://developers.cloudflare.com/d1/get-started/#4-run-a-query-against-your-d1-database), but that's too slow and complex to serve as an actual moderation tool.

To that effect I now have a very brutalist dashboard being served by the Webmentions server (code [here](https://git.msfjarvis.dev/msfjarvis/acceptable-vibes/src/branch/main/webmentions-server)) which has a list of all pages that have received Webmentions. To mask my inability to do good web design, it is styled using the [Oat](https://oat.ink) CSS library like all my non-blog web projects. I find the minimalist aesthetic of Oat quite nice and it rarely needs any significant changes to a semantically laid out page when you add it.

<Figure
  src={pages_with_webmentions_light}
  srcDark={pages_with_webmentions_dark}

Clicking on one of these links takes you to this equally terrible UI which could be reduced down to a list with delete buttons and still serve exactly the same purpose. Not exactly rocket science here.

<Figure
  src={received_webmentions_light}

I didn't want to figure out passwords and bearer auth, so the server does not attempt to do any authentication and just relies on Cloudflare Access to guard access to the UI and the endpoints this uses.

Hopefully I never have to use it!
