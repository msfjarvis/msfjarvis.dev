While reporting a bug to a project which was failing to run on my server, I had to go and dig up the systemd logs for it from \~18 hours ago which had since been buried by normal runtime logs (I had worked around the issue). Since I've been trying to wean myself off reaching for the LLM crutch all the time, here's a short rundown of my spelunking to find how to do this.

[This Unix StackExchange question](https://unix.stackexchange.com/questions/506833/show-journal-logs-from-the-time-a-service-was-restarted) introduce me to `journalctl INVOCATION_ID=$id + _SYSTEMD_INVOCATION_ID=$id` as a way to get logs for a specific "run" of a systemd service. Problem: `systemctl show -p InvocationID --value my-unit.service` shows the _current_ invocation ID which wasn't what I needed. I want an older one, when the service was failing.

I continued searching around, and came across [this excellent article](https://jwodder.github.io/kbits/posts/systemd-logs) which also explains how the incantation above actually works. To dig up historical invocation IDs, this shell pipeline can pull out start/stop events and collate them to invocation IDs — exactly what was needed here!

```plain
journalctl -o json -t systemd -u $service | jq -r '"[\(.__REALTIME_TIMESTAMP|tonumber / 1000000 | todate)] [\(.INVOCATION_ID)] \(.MESSAGE)"'
```

This is actually pretty simple when you break it down: get the systemd logs for the specified service in JSON format; then pull out the `.__REALTIME_TIMESTAMP`, `.INVOCATION_ID`, and `.MESSAGE` fields from each line. This let me get the logs I was looking for
